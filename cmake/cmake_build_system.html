3.2 Transitive Usage Requirements（可传递的使用要求）

Generally, a dependency should be specified in a use of target_link_libraries() with PRIVATE, INTERFACE and PUBLIC:

  • PRIVATE: dependency is used only in the implementation of a library, and not in the header files.
  • PUBLIC: dependency is used in both implementation and header files of a library.。
  • INTERFACE: dependency is used only in the header files of a library.

Usage requirements are propagated by reading the INTERFACE_ variants of target properties from dependencies and appending the values to the non-INTERFACE_ variants of the operand.

使用要求这样传递：目标属性中的 INTERFACE_ 变体取值从依赖中读取，然后将这些取值附加到相应属性的 non-INTERFACE_变体上。

In cases where order is relevant and maintained, and the order resulting from the target_link_libraries() calls does not allow correct compilation, use of an appropriate command to set the property directly may update the order.

如果 include 顺序维护有相关性，但 target_link_libraries()调用产生的顺序又不能正确编译，使用合适的命令直接设置属性可以更新顺序。

使用 install(export)命令导出为了安装的目标时，指定目标的使用要求时候必须小心。

3.4 Property Origin Debugging（面向调试的属性）

Because build specifications can be determined by dependencies, the lack of locality of code which creates a target and code which is responsible for setting build specifications may make the code more difficult to reason about.

由于构建规范可以由依赖决定，由于缺少创建目标和设置构建规范代码的局部性，导致代码更加难以推理。c

make 提供了一种调试机制，可以打印由依赖决定的属性的原始内容。将调试的属性在变量 CMAKE_DEBUG_TARGET_PROPERTIES 中列出。
